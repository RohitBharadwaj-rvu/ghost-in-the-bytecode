NON-FUNCTIONAL REQUIREMENTS
Ghost in the Bytecode

1. PERFORMANCE REQUIREMENTS
1.1 Latency
    - Injection and extraction operations for standard class files (< 100KB) and payloads (< 10KB) must complete in under 2 seconds on standard server hardware (2 vCPU, 4GB RAM).
1.2 Throughput
    - The system should support concurrent processing of at least 50 simultaneous requests without significant degradation in response time (> 500ms added latency).
1.3 Overhead
    - Attribute Mode: Injection overhead must be strictly limited to the payload size plus a minimal header (< 100 bytes).
    - S-Box Mode: Code generation overhead must not exceed 2KB per injected method to prevent class file bloat and potential JVM loading issues.

2. SAFETY AND SECURITY REQUIREMENTS
2.1 Zero Knowledge Architecture
    - The backend service must never receive or process plaintext payloads or user passphrases. All encryption and decryption must occur on the client-side.
2.2 Data Privacy
    - No payload data shall be logged or stored persistently on the server. Temporary files created during processing must be securely wiped immediately after the operation completes.
2.3 Integrity
    - All encrypted payloads must use AES-256-GCM authentication tags to detect tampering.
    - Modified carrier classes must pass valid JVM class verification (using ASM `CheckClassAdapter`) before being returned to the user.
2.4 Stealth
    - The injected bytecode in S-Box mode must be syntactically valid and unreachable or side-effect free, designed to evade heuristic analysis by standard antivirus software.

3. RELIABILITY AND AVAILABILITY
3.1 Error Handling
    - The system must provide distinct, user-friendly error messages for all failure scenarios (e.g., "Invalid Passphrase", "Corrupt Carrier File", "Payload Too Large"), rather than generic "Server Error" responses.
3.2 Availability
    - The API service should be designed for 99.9% uptime during business hours, with graceful degradation (e.g., queuing requests) under heavy load.
3.3 Fault Tolerance
    - A failure in one injection/extraction request must not affect the processing of other concurrent requests. The service must be stateless to allow for easy restart and scaling.

4. COMPATIBILITY AND PORTABILITY
4.1 Browser Compatibility
    - The frontend must fully support the latest versions of Chrome, Firefox, Edge, and Safari, specifically requiring support for the Web Crypto API (SubtleCrypto).
4.2 Backend Compatibility
    - The backend application must be compatible with Java 17 and higher (LTS versions) to ensure long-term support and security updates.
4.3 Bytecode Versioning
    - The system must support carrier classes from Java 8 (class file version 52) up to the latest Java version supported by the internal ASM library.

5. MAINTAINABILITY
5.1 Modularity
    - The codebase must strictly separate the `api` (controller), `engine` (core logic), and `common` (shared utilities) modules to facilitate independent updates and testing.
5.2 Documentation
    - All public API endpoints must be documented using OpenAPI/Swagger. Java code must adhere to Google Java Style Guide and include Javadoc for all public interfaces.
5.3 Testability
    - The core injection and extraction logic must be covered by unit tests with at least 80% code coverage.

6. USABILITY
6.1 User Feedback
    - The interface must provide immediate visual feedback for all user actions (e.g., loading spinners, progress bars, success notifications).
6.2 Accessibility
    - The web interface should follow WCAG 2.1 AA standards, ensuring sufficient color contrast and keyboard navigability.
6.3 Ease of Use
    - The primary workflows (Injection and Extraction) should require no more than 3 user interactions (Drag file -> Enter Passphrase -> Click Button).

7. SCALABILITY
7.1 Horizontal Scaling
    - The backend must be stateless, allowing for horizontal scaling by adding more server instances behind a load balancer to handle increased traffic.
7.2 Resource Management
    - The system must implement limits on file sizes (e.g., Max Carrier: 10MB, Max Payload: 1MB) to prevent memory exhaustion (OOM) attacks.
